## Feature Description

Phase F completes the config system with four features:

1. **`sickbay init --sync`** — Reads existing config, detects which applicable checks are missing, appends them as `true` without touching existing entries. Useful after `npm update sickbay` adds new runners.
2. **Scan-time notification** — When unlisted checks run (checks not in the config file), print a notice: "N new checks available but not in your config. Run `sickbay init --sync` to add them."
3. **`sickbay init --reset-config`** — Regenerates config from scratch, overwriting the existing file. Safety valve when config has drifted too far.
4. **Per-package `sickbay.config.ts`** — In monorepo mode, each package can have its own config that merges with (and overrides) the root config.

## User Story

As a Sickbay user maintaining a config file
I want to keep it in sync with new checks and customize per-package settings
So that I maintain full discoverability and per-package calibration as my project and Sickbay evolve

## Problem Statement

When Sickbay adds new runners in updates, existing config files miss them. The checks still run (unlisted = enabled), but users lose the discoverability benefit of seeing all checks in one place. In monorepos, a single root config can't express per-package calibration needs (e.g. different thresholds for a React app vs a Node API).

## Solution Statement

- `--sync` performs a non-destructive append of missing checks to the existing config file using AST-free text manipulation (find the closing `},` of the checks block, insert new entries before it)
- Scan-time warning compares runner names against config check IDs and emits a stderr notice
- `--reset-config` calls the existing `generateConfigFile` with a force flag to overwrite
- Monorepo config merges by loading both root and package configs, deep-merging with package taking precedence

## IMPLEMENTATION PLAN

### Phase 1: `sickbay init --sync` — Core Logic

Add a `syncConfigFile()` function to `apps/cli/src/commands/init.ts` that:

1. Reads the existing config file content as a string
2. Calls `detectContext()` + `getAvailableChecks()` to get all applicable check IDs
3. Parses which check IDs are already present in the file (regex scan for check name patterns — both `checkName:` and `'check-name':`)
4. Identifies missing checks (applicable but not in the file)
5. Groups missing checks by category (same `CATEGORY_ORDER` as `generateConfigFile`)
6. Finds the insertion point: the last `// --- Category ---` comment block or the closing `},` of the checks object
7. Inserts new checks with category comment headers, matching existing indentation
8. Writes the modified file back

**Key design decisions:**
- Text manipulation, not AST. The config file is simple and predictable (generated by us or hand-edited by users). Regex-based insertion is sufficient and avoids a heavy dependency.
- Missing checks are appended after existing checks, grouped by category with `// --- New: Category ---` headers to clearly delineate what was added.
- If the file has no `checks:` block at all, warn and skip (edge case: user has a weights-only config).

### Phase 2: `sickbay init --reset-config` — Force Overwrite

Modify `generateConfigFile()` to accept an optional `force` flag:

```typescript
export async function generateConfigFile(projectPath: string, options?: { force?: boolean }): Promise<void>
```

When `force` is true, skip the `existsSync` early return and overwrite the file.

### Phase 3: CLI Flag Wiring

Update the `init` command in `apps/cli/src/index.ts`:

```typescript
program
  .command('init')
  .description('Initialize .sickbay/ folder and run an initial baseline scan')
  .option('-p, --path <path>', 'project path to initialize', process.cwd())
  .option('--sync', 'add newly available checks to existing config without touching existing entries')
  .option('--reset-config', 'regenerate config from scratch (overwrites existing)')
  .action(async (options) => {
    if (options.sync) {
      const { syncConfigFile } = await import('./commands/init.js');
      await syncConfigFile(options.path);
      return;
    }
    if (options.resetConfig) {
      const { generateConfigFile } = await import('./commands/init.js');
      await generateConfigFile(options.path, { force: true });
      return;
    }
    const { initSickbay } = await import('./commands/init.js');
    await initSickbay(options.path);
  });
```

Note: `--sync` and `--reset-config` are quick operations — they don't run a baseline scan. Only bare `sickbay init` runs the full init flow.

### Phase 4: Scan-Time Notification

In `packages/core/src/runner.ts`, after resolving the applicable runners and loading the config, compare runner names against config check IDs:

```typescript
// In runSickbay(), after filtering runners and before running checks:
if (config?.checks) {
  const configCheckIds = new Set(Object.keys(config.checks));
  const unlistedRunners = runners.filter(r => !configCheckIds.has(r.name));
  if (unlistedRunners.length > 0) {
    process.stderr.write(
      `Note: ${unlistedRunners.length} check(s) running but not listed in your config: ` +
      `${unlistedRunners.map(r => r.name).join(', ')}. ` +
      `Run \`sickbay init --sync\` to add them.\n`
    );
  }
}
```

This only fires when a config file exists (`config !== null`) and has a `checks` block. Zero-config users never see it.

### Phase 5: Per-Package Config in Monorepo Mode

Modify `runSickbayMonorepo()` in `packages/core/src/runner.ts`:

1. Load root config as before: `const rootConfig = await loadConfig(rootPath)`
2. For each package, attempt to load a package-level config: `const pkgConfig = await loadConfig(pkgPath)`
3. If both exist, deep-merge with package overriding root
4. Pass the merged config via `_config` to `runSickbay()`

Add a `mergeConfigs()` function to `packages/core/src/config.ts`:

```typescript
export function mergeConfigs(root: SickbayConfig | null, pkg: SickbayConfig | null): SickbayConfig | null {
  if (!root && !pkg) return null;
  if (!root) return pkg;
  if (!pkg) return root;

  return {
    // Package checks override root checks per-key
    checks: { ...root.checks, ...pkg.checks },
    // Package exclude appends to root exclude
    exclude: [...(root.exclude ?? []), ...(pkg.exclude ?? [])],
    // Package weights override root weights per-category
    weights: root.weights || pkg.weights
      ? { ...root.weights, ...pkg.weights }
      : undefined,
  };
}
```

Merge semantics:
- **checks**: Per-key override. Package can disable a check the root enables, or set different thresholds. Uses spread so package entries win on collision.
- **exclude**: Additive. Package paths are appended to root paths. A package can't un-exclude a root-level pattern (this is the safe direction).
- **weights**: Per-category override. Package can adjust weight emphasis differently. If neither has weights, result is undefined (uses defaults).

Update `configMeta` on `MonorepoReport` to reflect whether root config, per-package configs, or both are active.

## NOTES

- **No web changes needed** — the ConfigTab already renders whatever config metadata the report contains. Per-package configs will flow through naturally since each `SickbayReport` already carries its own `config` field.
- **Sync is text-based, not AST-based** — this is intentional. The config file format is simple enough that regex insertion works reliably. If the user has a heavily customized config shape, `--sync` finding the insertion point may fail — in that case, print a warning and ask the user to add checks manually.
- **`--sync` vs `--reset-config`** — sync is additive (safe), reset is destructive (overwrites customizations). Both are explicit opt-in via flags.
- **Scan-time notice goes to stderr** — doesn't break `--json` output. Follows the existing pattern from `validateConfig()`.
