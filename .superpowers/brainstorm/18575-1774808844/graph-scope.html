What should the dependency graph show?

The graph needs a tree structure. Two very different data sources are possible.

A

npm Package Tree

Your project's npm dependencies and their transitive deps. Data from pnpm ls --json --depth 1.

my-app
lodash@4.17.20 (vulnerable)
react@18.2.0 (outdated)
express@4.18.2
├── loose-envify@1.4.0
├── body-parser@1.20.1
└── qs@6.11.0

Pros

  • Shows what you actually install
  • Highlights transitive vulnerability chains
  • Cross-references with outdated + npm-audit data
  • New data — complements the existing module graph

Cons

  • Needs new data source in core (pnpm ls)
  • Can be large even at depth 1
B

Enhanced Module Import Graph

The existing madge graph (file imports), but with dependency status overlaid on nodes. Already available in the report.

src/App.tsx
src/utils/api.ts
src/hooks/useData.ts
src/services/auth.ts (circular)

Pros

  • Data already exists in the report (madge metadata)
  • No new runner or core changes needed
  • Shows your code structure

Cons

  • Very similar to existing Codebase tab graph
  • Doesn't show npm packages at all
  • Can't highlight vulnerable/outdated deps
C

Hybrid: npm Package Tree (recommended)

Option A, but smart about it. Top-level deps only by default, with expand-on-click for transitive. Filter toggles to focus on problems. This gives the Dependencies tab something the Codebase tab doesn't have.

Why this one

  • Fills a real gap — no existing way to see the package tree
  • Annotations from existing runners make it immediately useful
  • Expand-on-click keeps it manageable
  • Distinct from the module graph on the Codebase tab
