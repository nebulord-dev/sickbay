# Vitals Config & Project Data Design

## Overview

Vitals needs a way for projects to calibrate checks to their own standards, and a place to store data it generates over time. This document covers the config file format, the `.vitals/` project folder, the `vitals init` command, and the related web UI surface.

---

## The Core Problem

Vitals ships with opinionated defaults (e.g. 300-line max per file, specific dep age thresholds). These make sense as universal starting points, but projects with established conventions shouldn't be penalized for not conforming to them. A codebase with 800-line components isn't necessarily unhealthy — it just has different standards.

The goal is **calibration**, not suppression. The user isn't dismissing the check; they're telling Vitals what "healthy" means for their project. Scores should reflect the project's health against its own declared standards, not a one-size-fits-all ruleset.

When custom thresholds are active, Vitals should surface a notice in both the TUI and web dashboard so the score is understood in context.

---

## Config File: `vitals.config.ts`

The config lives at the project root as `vitals.config.ts`, alongside `vite.config.ts`, `eslint.config.ts` etc. This is the established ecosystem pattern — visible, findable, consistent.

**Not** `.vitalsrc.json` — JSON gives no IDE support and no programmatic capability. **Not** `.vitals/config.ts` — config should be at the root where users expect it, not buried in a data folder.

Vitals executes it at runtime using `jiti` so users never need a build step.

### Shape

```ts
import { defineConfig } from 'vitals'

export default defineConfig({
  checks: {
    'file-complexity': {
      maxLines: 800,          // project default is 400
      maxCyclomaticComplexity: 20,
    },
    'dependency-age': {
      maxAgeDays: 365,        // project default is 180
    },
    'duplicate-code': {
      threshold: 30,          // % duplication before flagging
    },
  },

  // Exclude paths from all checks
  exclude: ['src/generated/**', 'src/legacy/**'],

  // Disable specific checks entirely
  disable: ['lighthouse'],

  // Override category weights (must sum to 1)
  weights: {
    security: 0.35,
    dependencies: 0.20,
    codeQuality: 0.25,
    performance: 0.10,
    git: 0.10,
  },
})
```

### `defineConfig` helper

Vitals exports `defineConfig` from `@vitals/core`. It's a typed identity function — no runtime logic, just TypeScript inference. The benefit is full autocomplete and validation in any editor without needing a JSON schema.

### Monorepo support

In a monorepo, `vitals.config.ts` at the workspace root applies to all packages. Individual packages can have their own `vitals.config.ts` for overrides. The per-package config merges with (and takes precedence over) the root config. This is deferred to Phase 3.

## Project Data Folder: `.vitals/`

Vitals generates data over time (history, cache). This lives in `.vitals/` at the project root — the same pattern as `.vscode/`, `.github/`, `.claude/`.

```
.vitals/
├── .gitignore          # generated by vitals init — sensible defaults
├── history/            # timestamped report snapshots
│   ├── 2026-03-01T14:32:00.json
│   └── 2026-03-04T09:15:00.json
├── cache/              # incremental check cache (speeds up re-runs)
└── baseline.json       # reference point for trend comparisons
```

### What gets gitignored by default

`vitals init` writes a `.vitals/.gitignore` with:

```
history/
cache/
```

`baseline.json` is **not** gitignored — committing it means the whole team shares the same reference point for score comparisons and CI diff checks.

Teams that want shared history (e.g. to track score over time in CI) can remove `history/` from the gitignore. That's an opt-in.

### `history/`

Each time `vitals` runs (or on an explicit `vitals snapshot` command — TBD), it appends a timestamped JSON report. This is the data source for the Historical Trends feature. Reports are the full `VitalsReport` JSON.

### `cache/`

Stores per-file hashes so incremental checks can skip unchanged files. Powers the 18x speedup described in the incremental checks backlog task. Gitignored — machine-local.

### `baseline.json`

A single committed report that serves as the canonical comparison baseline. Used by:
- `vitals diff` — compares current run against baseline
- CI — fails if score drops below baseline by a configurable threshold
- Web dashboard — shows delta from baseline alongside current score

## `vitals init` Command

Scaffolds everything needed to get started:

1. Creates `vitals.config.ts` at project root with all defaults shown as comments (so users can see what's available and uncomment what they want)
2. Creates `.vitals/` folder with a `.gitignore` inside
3. Runs an initial scan and saves the result as `.vitals/baseline.json`
4. Prints a summary of what was created

The generated `vitals.config.ts` is the primary discoverability surface for what's configurable. It should reference the scoring docs so users know what each threshold actually affects.

## Web UI: Config & Data Pages

### Config tab

A read-only view of the active configuration:

- Lists every check, its default threshold, and whether a custom value is in effect
- Highlights overridden values visually (e.g. a different color or "custom" badge)
- Shows which checks are disabled and which paths are excluded
- Links out to scoring docs for each check

**Editable config** (later): The CLI HTTP server could expose a write endpoint that proxies changes back to `vitals.config.ts` on disk. The UI would show a diff preview before writing. Not in scope for the initial implementation — read-only first.

### History tab (part of Historical Trends feature)

- Line chart of overall score over time, sourced from `.vitals/history/`
- Per-category trend lines, toggleable
- Baseline marker on the chart
- Clicking a point loads that report's full details

## Relationship to Other Features

| Feature | Dependency on this design |
|---|---|
| Historical Trends | Reads from `.vitals/history/` |
| Branch diff (`vitals diff`) | Reads `.vitals/baseline.json` |
| False positive suppression | `suppress` key in `vitals.config.ts` |
| `.vitalsrc` (original task) | Superseded by this design |
| CI/CD integration | References `baseline.json` for threshold enforcement |
| Incremental checks | Uses `.vitals/cache/` |
| Monorepo per-package config | Extends this schema in Phase 3 |

---

## What's Not Configurable

Some checks are objective and should not be threshold-configurable:

- Hardcoded secrets / credentials
- Known CVEs from `npm audit`
- Circular dependencies (existence, not count)

These can be suppressed on a case-by-case basis (see False Positive Suppression backlog task) but their scoring logic isn't user-adjustable.

---

## Implementation Notes

- `jiti` executes `vitals.config.ts` at runtime without requiring a user build step
- Config is loaded once in `runner.ts` and passed through as part of the runner context
- Each runner declares which config keys it reads — unknown keys should warn, not silently ignore
- The "custom thresholds in effect" notice should appear: in the TUI score panel, in the web dashboard header, and in `--json` output as a top-level flag
- Phase 3 will extend this schema for per-package monorepo overrides — design the config type to be extensible from the start
